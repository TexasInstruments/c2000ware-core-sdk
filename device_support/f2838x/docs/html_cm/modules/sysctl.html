SYSCTL Module¶

System Control (SysCtl) determines the overall operation of the communication manager (CM). The API provides functions to configure the set of peripherals that are enabled, the windowed watchdog, the NMI watchdog, and WFI. It also provides functions to handle and obtain information about resets and peripheral clock requests.

group sysctl_api

The System Control module is used for miscellaneous system configurations.

uint32_t SysCtl_getResetCause(void)¶

Gets the reason for a reset.

This function will return the reason(s) for a reset. Since the reset reasons are sticky until either cleared by software or an external reset, multiple reset reasons may be returned if multiple resets have occurred. The reset reason will be a logical OR of

  • SYSCTL_CAUSE_PORESETN - Power-on reset

  • SYSCTL_CAUSE_XRSN - External reset pin

  • SYSCTL_CAUSE_CPU1WDRSN - Watchdog reset from CPU1

  • SYSCTL_CAUSE_CPU1NMIWDRSN -NMI Watchdog reset from CPU1

  • SYSCTL_CAUSE_CPU1SYSRSN - System reset from CPU1

  • SYSCTL_CAUSE_CPU1SCCRESETN - SCCRESETn from CPU1

  • SYSCTL_CAUSE_ECAT_RESET_OUT - ECAT reset

  • SYSCTL_CAUSE_CPU1SIMRESET_CPURSN - SIMRESET from CPU1

  • SYSCTL_CAUSE_CMRSTCTLRESETREQ - System reset request from CM4

  • SYSCTL_CAUSE_CMVECTRESETN - Vectreset from CM4

  • SYSCTL_CAUSE_CMSYSRESETREQ - System reset request from CPU1

  • SYSCTL_CAUSE_CMNMIWDRSTN - NMI watchdog reset from CM4

  • SYSCTL_CAUSE_CMEOLRESETN - EOLRESETn reset from CM4

Return

Returns the reason(s) for a reset.

void SysCtl_clearResetCause(uint32_t rstCauses)¶

Clears reset reasons.

This function clears the specified sticky reset reasons. Once cleared, another reset for the same reason can be detected, and a reset for a different reason can be distinguished (instead of having two reset causes set). If the reset reason is used by an application, all reset causes should be cleared after they are retrieved with

SysCtl_getResetCause().
Parameters

  • rstCauses: are the reset causes to be cleared; must be a logical OR of SYSCTL_CAUSE_PORESETN, SYSCTL_CAUSE_XRSN, SYSCTL_CAUSE_CPU1WDRSN,SYSCTL_CAUSE_CPU1NMIWDRSN, SYSCTL_CAUSE_CPU1SYSRSN ,SYSCTL_CAUSE_CPU1SCCRESETN, SYSCTL_CAUSE_ECAT_RESET_OUT , SYSCTL_CAUSE_CPU1SIMRESET_CPURSN, SYSCTL_CAUSE_CMRSTCTLRESETREQ ,SYSCTL_CAUSE_CMVECTRESETN, SYSCTL_CAUSE_CMSYSRESETREQ, SYSCTL_CAUSE_CMNMIWDRSN and/or SYSCTL_CAUSE_CMEOLRESETN.

Return

None.

void SysCtl_delay(uint32_t count)¶

Delays for a fixed number of cycles.

This function generates a constant length delay using assembly code. The loop takes around 6 cycles per iteration plus 9 cycles of overhead.

Parameters

  • count: is the number of delay loop iterations to perform.

It is important to note that this function is a fairly accurate timing mechanism. Although the delay loop is deterministic, the execution time of the loop will vary dramatically depending upon the application’s interrupt environment (the loop will be interrupted unless run with interrupts disabled and this is generally an unwise thing to do) and also the current system clock rate and flash timings (wait states and the operation of the prefetch buffer affect the timing).

For best accuracy, a system timer should be used with code either polling for a particular timer value being exceeded or processing the timer interrupt to determine when a particular time period has elapsed.

Return

None.

Many of the functions provided by the SysCtl API are related to CM configuration. The ability to enable (turn on the module clock), disable (gate off the module clock), and perform a software reset on most of the peripherals on a device is provided by SysCtl_enablePeripheral(), SysCtl_disablePeripheral(), and SysCtl_resetPeripheral() respectively.

The CM’s windowed watchdog is enabled and disabled by SysCtl_enableWatchdog() and SysCtl_disableWatchdog() respectively. The watchdog can be serviced by SysCtl_serviceWatchdog(). Several functions are also provided to configure the watchdog’s clock and windowed functionality.

This section will give further details of these functions and each of the others used for the configuration of SysCtl.

The code for this module is contained in driverlib_cm/sysctl.c, with driverlib_cm/sysctl.h containing the API declarations for use by applications.
